SOClass Work Documentation.

 


Sunrise is software product designed to become a leading force in the emerging market of electronic trade information systems, with a special focus on international trade and operations of customs administrations.

The product is designed to operate globally in the Internet environment and fully applies the concept “the computer IS the network”. The programming language is Java, a fully object oriented language, which guarantees both an open environment an d an unrivaled potential of evolution. Java is currently the thriving force of companies like SUN, IBM, HP, Netscape, Oracle, etc. and in addition to his ever growing presence in PCs, Web browsers and network computers, is about to invade electronic appli ances, from TVs, phones, watches to cars electronics and smart homes.

Sunrise is composed of a core system (Kernel) and some standard application systems, from which specific Sunrise applications will be derived. The first standard applications will concern customs administration systems, national and internationa l transit-monitoring systems, and trade systems.

Sunrise users will essentially manipulate electronic documents that will be processed according to the rules of the kind of business they are dealing with. Some documents will be processed through a unique application A, whilst other will be processed by different applications A, B, etc.

Sunrise has the ability to transfer rights from the holder of a document to a new holder, and thus, replicates the tradit ional paper based system.

The Sunrise mission is to offer IT services (in the virtual world) replacing paper based services (in the real world) nec essary to complete a trade transaction: A trade transaction starts and ends in the real world but the services in-between disappear from the real world. The most obvious benefits are the processing speed and the savings made on the real world services. Th e other benefits lay in a very accurate and consistent enforcement of the various business and legal rules, and automated transfers of data between computers, that eliminate expensive and error prone re-entry procedures.
 
 

 Sunrise Glossary


 


A. User oriented

Access Group: Set of authorized actions per document implemented.

Action: Generic operation activated by the user, which modify the status of a document. For example, “Validate” is an action that changes the status of a document from “draft” to “validated”. An action results into performing a t ransaction on the server.

Billing profile: Set of parameters defining how the cost of the service is calculated, and when and where the service is invoiced.

BRL: Business rules and logic

Client: Computer, or network computer running the GUI

Customer: User or company employing users. A customer must register to a WSP. A Customer has a billing profile.

DBS: Database system

Desktop: The user's workplace, which can contain folders, documents and a wizard. The desktop is displayed after a successful login to a WSP. The user can customize the desktop, e.g. change color, themes, look and feel, etc. Thes e personal settings are automatically saved by the server at logoff, and restored at login.

Document: Frame containing one or many forms, each form containing one or many pages. The forms of a same document are linked together through the application logic.

Folder: Frame containing other folders or documents.

GUI: Graphic user interface

Main-Folder: Folder that contains the main Mail-IN, Mail-OUT, New-Document folders and the Finder/Wizard.

Server: Computer(s) running the BRL

Task: Elementary treatment of a document performed by the server.

Transaction: Treatment performed by the server when the user requests an action. A transaction is a linked list of (successful) tasks.

User: Physical person that has a login user Id and password. A user belongs to an access group and has a billing profile. A user is a customer.

Wizard: Character that pops up whenever the user is in trouble and smartly offers to assist.

Sunrise group: Corporation that develops the Sunrise kernel system and Sunrise standard application library, and deliver adequate training and qualifications to WSIs.

Sunrise service provider or WSP: Organi zation or corporatio n that operates a Sunrise service i.e. registers customers, runs the server computer( s) that deliver the service, and eventually invoices users. Often the WSP will also own the database.

Sunrise system integrator or WSI: Company that develops specific software a pplications derived from Sunrise standard application library. The system integrator is usually a software house that received an adequate training and passed the Sunrise qualification.

B. Developer oriented

Binder: Abstract class that binds together one or many instances of bind-able objects like Document, Folder, DataField, BinderShortcut, that will be retrieved from the server and displayed on the desktop.

BinderShortcut: Class that holds a link to a Document or Folder object.

Broker: The Broker Class, descendant of UnicastRemoteObject, provides the client with methods that match the actions requested. It also implements the per action authorization mechanism.

Data: Class that holds a reference to any kind of internal data. The Data class is the ancestor of most of non-visual classes used by the client tier. Each instance of Data object is connected to a set of Rule objects.

DataField: Terminal class, descendant of Data. Each instance is connected to a VisualFacet object.

DataSet: This class, descendant of Data, holds a flat vector of Data objects. DataSet being a descendant of Data, a DataSet object will usually hold a multi-level, tree-like, structure of Data.

Data-tree: The internal data representation of a document is a hierarchical tree-like one. Tree nodes are descendants of DataSet classes, and tree leaves are descendants of Data, but not descendants of DataSet classes.

Document: Class that holds the data structure representing logically the document, and the navigation logic to move from a form page to another one.

Folder: Container class containing BinderShortcut objects.

Page: Generic name used to designate a terminal class, descendant from JPanel, that defines color, position and size of entry fields and drawings. Each instance of Page knows how to display one page of a document for data entry or data di splay purposes.

Server: The Server is a descendant of UnicastRemoteObject. There is one instance of Server that is started at the beginning of a server session. The Server object first instances one SpecificServer object and then waits for clients to req uest a connection, checks upon user identification (user Id and password), and instances for each logged in client a Broker object.

SpecificServer: There is one instance of SpecificServer per application. It implements the corresponding services. It also control access authorizations.

ServerBinder: There is an instance of ServerBinder object created for each Document object handled by the SpecificServer class. The role of this object is a) to facilitate the access to those documents by the client, and b) to hold a set of ServerRule objects that will be applied to the Data-tree.

ServerRule: List of tasks that are applied on the data hold by a ServerBinder.

TableConnector: Class that connects internal data components to JDBC/SQL fields in the DBS. Transfer sdsd

Visual-object-X: Generic name for visual objects. Each instance of a Visual-object-X holds a reference to an instance of an object-X. This allows for a total distinction between the programming of visual and non-visual behaviors. In addit ion, many instances of Visual-object-X objects can be connected to a same instance of a X object.

VisualBinder: Visible frame that contains a Document or a Folder.

VisualBinderShortcut: Visible icon that holds a link to a Document or Folder through a BinderShortcut.

VisualDocument: Visible set of Form objects. The Document defines the order of the forms through a link to a Document object.

VisualForm: Visible class that contains one or many pages. The VisualForm objects defines the sequence of the pages, and which page class is used when creating an instance of a new page in the form.

VisualFolder: Visible class that holds a link to a Folder class and implements a panel containing VisualBinderShortcuts. VisualFolder objects support both the list and icon models.

VisualFacet: Invisible abstract class that connects internal data components to fields in the pages of documents. The actual connection is made through descendants classes of VisualFacet that implement specific links with specific visual controls like text, date, number, combo box, etc.
